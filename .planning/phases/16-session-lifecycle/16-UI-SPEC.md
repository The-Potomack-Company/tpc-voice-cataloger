## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (`gap-1`, `mt-1`) | Badge gaps, inline vertical nudges |
| sm | 8px (`gap-2`, `py-2`, `px-2`, `mb-2`, `space-y-2`) | Compact element spacing, badge padding, card-to-card spacing |
| md | 16px (`p-4`, `px-4`, `mt-4`) | Default card/banner padding, page horizontal padding (portrait) |
| lg | 24px (`gap-6`, `py-6`, `mb-6`) | Page vertical padding, section bottom margin, banner bottom margin |
| xl | 32px (`mb-8`, `px-8`, `mt-8`) | Section breaks between lifecycle status sections, landscape horizontal padding |
| 2xl | 48px (`min-h-12`) | Touch target minimum height |
| 3xl | 64px | Not used in this phase |

Exceptions:
- `min-h-12` (48px) on all interactive buttons for thumb-zone accessibility (UX-02 requirement).
- Inherited `mb-3` (12px) on existing section headers and `space-y-3` (12px) between cards -- these are pre-existing off-grid values from prior phases. All new elements introduced in Phase 16 use on-grid values only.

Source: Consistent with Phase 15 UI-SPEC spacing scale. Existing `Sessions.tsx`, `SessionDetail.tsx`, `SessionCard.tsx` patterns.

| Section | Header Treatment |
|---------|-----------------|
| Needs Attention (specialist, returned sessions) | `text-amber-600 dark:text-amber-400` with warning triangle icon (inline SVG, `w-4 h-4` before text) |
| Awaiting Review (admin, submitted sessions) | Standard section header: `text-gray-500 dark:text-gray-400` (no special color -- admin's primary work queue is distinguished by position at top, not color) |

| Returned (admin view) | Standard: `text-gray-500 dark:text-gray-400` |

Rationale: Only the specialist's "Needs Attention" section uses color emphasis because it represents sessions that require the specialist's action -- it needs to draw the eye. ### Return Dialog Button Color

The "Return Session" confirm button uses `bg-amber-600 hover:bg-amber-700 text-white` rather than the accent blue. Rationale: Returning a session is a cautionary action (not destructive, not primary navigation). Amber communicates "proceed with awareness" and matches the returned-session visual theme.

Source: `--color-accent: #2563eb` from `src/index.css`. Status badge colors from Phase 15 UI-SPEC. Banner colors from RESEARCH.md code examples, cross-referenced with existing interrupted-recording banner in `SessionDetail.tsx`.

## Visual Hierarchy & Focal Point

### Sessions Page -- Specialist View

The specialist Sessions page now has role-aware lifecycle sections. Visual scanning hierarchy:

1. **Primary focal point: "Needs Attention" section with amber header.** The amber-colored section header with warning icon draws the eye first. This section contains returned sessions requiring specialist action -- the highest-priority items.
2. **Secondary focal point: Active sessions.** The specialist's current in-progress work. Standard section header styling.
3. **Tertiary focal point: Status pills on non-active cards.** "Submitted" (yellow) and "Exported" (green) pills provide at-a-glance status within the Submitted and Exported sections.

### Sessions Page -- Admin View

1. **Primary focal point: "Awaiting Review" section at top.** Position (first rendered section) makes it the primary work queue. Contains submitted sessions grouped by specialist (Phase 15 grouping pattern).
2. **Secondary focal point: Status badges on SessionCards.** Colored pills provide rapid status scanning.
3. **Tertiary focal point: "Returned" section.** Sessions sent back to specialists, waiting for their response.

### SessionDetail Page -- Submitted Session (Specialist)

1. **Primary focal point: Blue status banner.** "Submitted -- awaiting admin review" banner with clock icon immediately below the header communicates the locked state.
2. **Secondary focal point: Absence of interactive affordances.** Submit button gone, Add Item button gone, EditableField displays as plain text (no hover cursor), RecordButton hidden. The absence of interactivity reinforces the read-only state.

### SessionDetail Page -- Returned Session (Specialist)

1. **Primary focal point: Amber sticky banner with review notes.** Pinned at top of scroll area, always visible. Warning icon + "Returned by Admin" heading + review notes text. This is the most important element -- it tells the specialist what to fix.
2. **Secondary focal point: Restored editing affordances.** Submit button reappears, Add Item button returns, fields become tappable again. The contrast with the previously-locked state signals "you can work on this again."

### 6. Read-Only State Visual Communication

When a session is submitted and the user is a specialist, all editing affordances are visually suppressed:

| Component | Active/Returned State (Editable) | Submitted State (Read-Only) |
|-----------|----------------------------------|----------------------------|
| `EditableField` | Tappable text with `cursor-pointer hover:bg-gray-100` highlight | Static text, no cursor change, no hover highlight. Uses existing `readOnly` prop which renders text without click handler. |
| `RecordButton` | Red mic button visible | Hidden entirely (`{!isReadOnly && <RecordButton />}`). Not disabled, not grayed -- completely absent. |
| Floating "Add Item" | Blue accent button fixed at bottom | Hidden entirely (`{!isReadOnly && <div>...</div>}`). |
| Session name | Tappable with `cursor-pointer hover:text-accent` | Static text, no cursor, no hover. Uses existing `isReadOnly` check. |
| Notes textarea | Editable `<textarea>` | Read-only `<div>` with `bg-gray-50` background. Uses existing `isReadOnly` conditional. |
| "Submit for Review" button | Visible for specialist on active session | Hidden (session is no longer active). |
| "Mark Complete" / "Reopen" buttons | No longer rendered | Replaced by lifecycle buttons in Phase 16. |

**Read-only derivation logic:**
```typescript
const { isAdmin } = useUserRole();
const isSpecialist = !isAdmin;
const isLifecycleLocked = isSpecialist && session.status === 'submitted';
const isReadOnly = isLifecycleLocked;
// Admin is NEVER locked by session status for editing purposes
```

## Interaction States

### Loading States

| Scenario | Visual Treatment |
|----------|-----------------|
| Role detection (Sessions page) | Search bar visible immediately; below it, 3 skeleton cards (Phase 15 pattern). No lifecycle sections until role resolves. |
| Role detection (SessionDetail page) | Page renders with content visible; lifecycle buttons area empty until role resolves. Read-only defaults to `true` (conservative) until role is known. |
| Submit action in progress | "Submit for Review" button disabled with `opacity-50`, text unchanged. Optimistic update shows status banner immediately. |
| Return action in progress | ReturnDialog "Return Session" button disabled with `opacity-50`. Dialog closes on success. |
| Export in progress | Spinning icon replaces download icon. Button text changes to "Exporting...". Button disabled. |

### Error States

| Scenario | Visual Treatment |
|----------|-----------------|
| Submit fails | Optimistic update reverts: status returns to `active`, status banner disappears, Submit button reappears, fields become editable again. Console error logged. No user-facing error toast (consistent with existing optimistic revert pattern). |
| Return fails | Optimistic update reverts: status returns to `submitted`. ReturnDialog closes. Console error logged. |
| Export download fails | Button re-enables, text returns to "Export Session". Status does NOT change to `exported` (status update only runs after successful download). Console error logged. |
| Role detection fails | Falls back to specialist view (safe restrictive default). Console error logged. |

### Optimistic Updates

| Action | Optimistic Behavior |
|--------|-------------------|
| Submit session | Session status updates to `'submitted'` in Zustand store immediately. If Supabase write fails, reverts to `'active'`. |
| Return session | Session status updates to `'returned'` and `review_notes` updates in store immediately. If Supabase write fails, reverts to `'submitted'`. |
| Export + status update | JSON downloads first (no optimistic update for download). After download succeeds, status updates to `'exported'` optimistically. If status write fails, reverts to previous status. |

## Accessibility

| Concern | Implementation |
|---------|----------------|
| Touch targets | All buttons use `min-h-12` (48px) per UX-02. Submit, Return, Export, Delete buttons all meet this minimum. |
| Submit confirmation | Uses existing `ConfirmDialog` which is keyboard-accessible (focusable buttons, Escape to dismiss). |
| ReturnDialog | `<textarea>` is natively keyboard-accessible. Dialog uses `createPortal` to body for correct stacking. Focus should be moved to textarea on open. |
| Status banners | Banners use informative color + icon + text. Not reliant on color alone -- text content ("Submitted -- awaiting admin review") conveys the state. |
| Read-only state | `EditableField` in read-only mode removes the click handler entirely -- screen readers will not announce it as interactive. RecordButton is removed from DOM, not just hidden. |
| Section headers | "Needs Attention" section uses text + icon (not color alone) to convey urgency. |
| Status pills | Pill text labels ("Submitted", "Returned", "Exported") convey meaning without relying on color. |
| Color contrast | Blue-700 on blue-50, amber-800 on amber-50, yellow-700 on yellow-100, orange-700 on orange-100, green-700 on green-100 all pass WCAG AA. |
| Dialog backdrop | `bg-black/50` provides sufficient contrast. Dialog content is the only interactive area when open. |
