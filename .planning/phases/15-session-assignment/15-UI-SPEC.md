## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (`gap-1`, `mt-1`) | Badge gaps, inline vertical nudges |
| sm | 8px (`gap-2`, `py-2`, `px-2`) | Compact element spacing, badge padding, chevron-to-text gap |
| md | 16px (`p-4`, `px-4`) | Default card padding, page horizontal padding (portrait) |

Exceptions:
- `gap-3` (12px) is used between session cards within a section (`space-y-3`) and between specialist group content -- established project pattern.
- `min-h-12` (48px) on all interactive elements for thumb-zone accessibility (UX-02 requirement).
- `mb-3` (12px) below section headers and specialist group headers -- established project pattern.
- `mt-6` (24px) above the first section (Active Sessions) after the search bar.
- `mt-4` (16px) between specialist groups within a section.

Source: Existing `Sessions.tsx`, `SessionCard.tsx`, `NewSession.tsx`, `SessionDetail.tsx` patterns.

## Component Inventory

**Focal point:** The primary focal point is the session list on the Sessions page -- this is where admin and specialist roles diverge visually. The admin sees specialist-grouped collapsible sections with extended session cards; the specialist sees the unchanged flat list. Secondary focal points are the specialist dropdown on NewSession (admin-only) and the assignee field on SessionDetail (admin-only).

### 1. Sessions Page -- Admin View (`Sessions.tsx`)

**Layout:** Same page shell as current -- `portrait:px-4 landscape:px-8 landscape:max-w-3xl landscape:mx-auto py-6`. Same `SessionSearch` component at top.

**Role detection:** Use `useUserRole()` hook (queries `profiles.role` for current user). Show loading state until role is resolved. Then render admin view or specialist view based on result.

**Loading state (role detection):** Search bar renders immediately. Below it, show 3 skeleton cards: `bg-white dark:bg-gray-800 rounded-xl p-4 h-[72px] animate-pulse border border-gray-200 dark:border-gray-700` with `space-y-3` between them.

**Admin view structure:**
```
SessionSearch
[Active Sessions (N)]                           -- section header (always visible, not collapsible)
  [Specialist Name (M)]                         -- collapsible group, expanded by default
    SessionCard (admin variant)
    SessionCard (admin variant)
  [Specialist Name (M)]                         -- collapsible group, expanded by default
    SessionCard (admin variant)
[Completed (N)]                                 -- collapsible section (existing chevron pattern)
  [Specialist Name (M)]
    SessionCard (admin variant)
[Archived (N)]                                  -- collapsible section (collapsed by default)
  [Specialist Name (M)]
    SessionCard (admin variant)
```

**Specialist view structure:** Identical to current `Sessions.tsx` -- flat list per section, no grouping, no assignee info, no status badges. No changes whatsoever.

### 2. Specialist Group Header (Admin View)

**Purpose:** Collapsible header within each status section that groups sessions by assignee.

**Element:** `<button>` (same as existing Completed/Archived toggles -- keyboard-accessible).

**Classes:**
```
flex items-center gap-2 mb-3 ml-1
```

The `ml-1` (4px) provides a subtle indent to visually distinguish the specialist sub-group from the parent section header.

**Chevron icon:** Same SVG as existing collapsible sections in Sessions.tsx (lines 159-167).
```
className={`w-4 h-4 text-gray-500 dark:text-gray-400 transition-transform ${expanded ? 'rotate-90' : ''}`}
```
SVG: `<path strokeLinecap="round" strokeLinejoin="round" d="M9 5l7 7-7 7" />`

**Label:**
```html
<span class="text-sm font-semibold text-gray-500 dark:text-gray-400 uppercase tracking-wider">
  Sarah (3)
</span>
```

**Default state:** All specialist groups expanded by default (CONTEXT.md locked decision).

**Toggle behavior:** Click to collapse/expand with chevron rotation. Same implementation as existing Completed/Archived toggles.

**Content spacing:** `space-y-3` between session cards within the group (matches existing).

**Group spacing:** `mt-4` between specialist groups within a section.

**Group ordering:** Specialist groups ordered alphabetically by display name within each section.

**Edge case -- unassigned sessions:** If any sessions have `assigned_to = null` (should not happen per decisions, but defensive), show them under an "Unassigned" group header at the bottom of each section.

**Edge case -- empty group:** Do not render a group header if a specialist has 0 sessions in that section.

### 3. SessionCard -- Admin Variant (`SessionCard.tsx`)

**Extended props:**
```typescript
interface SessionCardProps {
  session: Session;
  itemCount: number;
  onTap: () => void;
  onDelete: () => void;
  onRename: (newName: string) => void;
  assigneeName?: string;      // NEW: shown for admin view only
  sessionStatus?: string;     // NEW: 'active'|'submitted'|'returned'|'exported', admin view only
}
```

**Metadata row extension:** The existing metadata row below the session name (`flex items-center gap-2 mt-1 flex-wrap`, line 113) is extended with two new elements when the optional props are provided.

**Assignee name placement:** After the item count span, before any status badge. Shown as plain text (not a badge pill).
```html
<span class="text-xs text-gray-500 dark:text-gray-400">
  Assigned to Sarah
</span>
```

The prefix "Assigned to" is included for clarity in the list scanning context. The `text-xs text-gray-500` matches the existing item count styling on line 117.

**Status badge placement:** After the assignee name, inline in the same `flex-wrap` row. Uses status-specific colors from the Color section.
```html
<span class="inline-flex items-center text-xs font-medium px-2 py-0.5 rounded-full bg-blue-100 dark:bg-blue-900/30 text-blue-700 dark:text-blue-400">
  Active
</span>
```

**Status badge label mapping:**

| Status value | Badge label | Badge color |
|-------------|------------|-------------|
| `active` | "Active" | Blue |
| `submitted` | "Submitted" | Yellow |
| `returned` | "Returned" | Orange |
| `exported` | "Exported" | Green |

**When `sessionStatus` is provided:** The new status badge replaces the existing "Completed" and "Archived" badges (lines 120-141) since those were v1.0 local-only statuses. The admin view uses the server-authoritative status from the `sessions.status` column.

**Specialist view:** When `assigneeName` and `sessionStatus` are not provided (undefined), the card renders exactly as today -- no visual changes for specialist users.

### 4. NewSession Page -- Specialist Dropdown (`NewSession.tsx`)

**Condition:** Dropdown renders only when the current user is admin (detected via `useUserRole()`). Specialists do not see this field.

**Placement:** Between the Notes textarea and the Import Receipt List / Submit button. Same `mb-6` spacing as other form fields.

**Label:**
```html
<label class="block mb-1 text-sm font-medium text-gray-700 dark:text-gray-300">
  Assign To
</label>
```

**Dropdown element:** Native HTML `<select>` (CONTEXT.md locked decision).

**Dropdown classes:**
```
w-full px-4 py-3 rounded-lg bg-gray-100 dark:bg-gray-800 text-gray-900 dark:text-gray-100
border border-gray-200 dark:border-gray-700
focus:outline-none focus:ring-2 focus:ring-accent min-h-12
appearance-none
```

These classes match the existing text input styling in `NewSession.tsx` (session name input, line 107). The `appearance-none` ensures consistent cross-browser styling.

**Select wrapper (for custom chevron):**
```html
<div class="relative mb-6">
  <select class="...">...</select>
  <svg class="pointer-events-none absolute right-3 top-1/2 -translate-y-1/2 w-5 h-5 text-gray-400 dark:text-gray-500"
       fill="none" viewBox="0 0 24 24" stroke="currentColor" strokeWidth={2}>
    <path strokeLinecap="round" strokeLinejoin="round" d="M19.5 8.25l-7.5 7.5-7.5-7.5" />
  </svg>
</div>
```

**Options:**
- First option: `<option value="" disabled>Select a specialist...</option>` (placeholder, disabled, selected by default)
- Then: all active accounts from `listAccounts().filter(a => a.is_active)`
- Display name as option text, account ID as option value
- Admin appears in the list alongside specialists (CONTEXT.md locked decision)
- Accounts sorted alphabetically by display name
- The form submit button is disabled until a valid specialist is selected

**Loading state:** While accounts are loading, show a disabled select with a single option "Loading..." and `opacity-50` on the wrapper.

**Error state:** If `listAccounts()` fails, show error text below the dropdown: `text-sm text-red-600 dark:text-red-400 mt-1` with copy "Could not load team members. Check your connection and try again." Submit button remains disabled.

**Specialist auto-assignment:** When the current user is a specialist, the dropdown is not rendered. The `createSession` call sets `assigned_to: user.id` automatically (no UI needed).

### 5. SessionDetail Page -- Assignee Field (`SessionDetail.tsx`)

**Condition:** Rendered only when the current user is admin (detected via `useUserRole()`). Specialists do not see this field.

**Placement:** Below the badge row (mode/status/archived badges at line 257) and above the item list. Wrapped in a `mb-6` container to match spacing rhythm.

**Display mode container:**
```html
<div class="flex items-center justify-between bg-gray-50 dark:bg-gray-800 rounded-lg px-4 py-3 mb-6">
  <span class="text-sm text-gray-500 dark:text-gray-400">Assigned to</span>
  <span class="text-sm font-medium text-gray-900 dark:text-gray-100 cursor-pointer hover:text-accent transition-colors"
        title="Tap to reassign">
    Sarah
  </span>
</div>
```

The `bg-gray-50 dark:bg-gray-800 rounded-lg` container provides visual distinction from the surrounding content, making the field feel like a metadata row. This container style matches `AccountRow.tsx` and Settings page row patterns.

**Tap-to-edit behavior:** Tapping the assignee name transitions to a native `<select>` dropdown inline, following the `EditableField.tsx` pattern (click to edit, blur to save/cancel).

**Editing mode container:**
```html
<div class="flex items-center justify-between bg-gray-50 dark:bg-gray-800 rounded-lg px-4 py-3 mb-6">
  <span class="text-sm text-gray-500 dark:text-gray-400">Assigned to</span>
  <select autoFocus
    class="text-sm rounded border border-gray-300 dark:border-gray-600
           bg-white dark:bg-gray-700 text-gray-900 dark:text-gray-100
           px-2 py-1 focus:outline-none focus:ring-2 focus:ring-accent">
    <option value="uuid-1">Sarah</option>
    <option value="uuid-2">Mike</option>
  </select>
</div>
```

**Save behavior:** Selecting a different user triggers an immediate Supabase update (`onChange`). No confirmation dialog (CONTEXT.md locked decision -- reassignment is low-risk and easily reversible). The dropdown closes on `change` or `blur`.

**Error handling:** If the update fails, revert to the previous assignee name and show an inline error below the field: `text-sm text-red-600 dark:text-red-400 mt-1` with copy "Reassignment failed. Please try again." Error auto-clears after 5 seconds.

**Dropdown options:** Same list as NewSession dropdown -- all active accounts from `listAccounts()`, filtered for `is_active`, sorted alphabetically. Current assignee is pre-selected.

### 6. useUserRole Hook (`src/hooks/useUserRole.ts`)

**Purpose:** Reusable hook for role detection. Extracted from the pattern already used in `AdminRouteGuard.tsx`.

**Return type:** `{ role: string | null; isAdmin: boolean; loading: boolean }`

**Behavior:**
- On mount, queries `profiles` table for current user's role
- Returns `loading: true` until query completes
- Returns `isAdmin: true` when `role === 'admin'`
- On query failure, returns `role: null, isAdmin: false` (safe default -- specialist view)
- Caches result for component lifetime (no re-fetch on re-render)

**Used by:** `Sessions.tsx`, `NewSession.tsx`, `SessionDetail.tsx`

## Interaction States

### Loading States

| Scenario | Visual Treatment |
|----------|-----------------|
| Role detection (Sessions page) | Search bar visible immediately; below it, 3 skeleton cards: `bg-white dark:bg-gray-800 rounded-xl p-4 h-[72px] animate-pulse border border-gray-200 dark:border-gray-700` with `space-y-3` |
| Role detection (NewSession page) | Form fields render normally; dropdown area is empty until role resolves (no flash) |
| Role detection (SessionDetail page) | Page renders normally; assignee row is absent until role resolves (appears after load) |
| Specialist list loading (NewSession dropdown) | Disabled `<select>` with "Loading..." option, `opacity-50` wrapper |
| Specialist list loading (SessionDetail edit) | Accounts load only when user taps to edit; display name shown as text until then |
| Reassignment in progress | Name updates optimistically to new selection; reverts on error |
| Creating session (existing) | "Creating..." button text, button disabled (no change from current behavior) |

### Error States

| Scenario | Visual Treatment |
|----------|-----------------|
| Role detection fails | Fall back to specialist view (show user's own sessions only -- safe restrictive default). Log error to console. |
| Specialist list fails to load (NewSession) | Error text below dropdown: `text-sm text-red-600 dark:text-red-400 mt-1` -- "Could not load team members. Check your connection and try again." Submit button disabled. |
| Specialist list fails to load (SessionDetail) | Assignee name displays normally (read-only). Tapping does nothing. No error shown. |
| Reassignment fails | Revert to previous assignee name. Inline error below the field: `text-sm text-red-600 dark:text-red-400 mt-1` -- "Reassignment failed. Please try again." Auto-clears after 5 seconds. |
| Session list fails to load | Handled by Phase 14 data layer -- not introduced in this phase |

### Optimistic Updates

| Action | Optimistic Behavior |
|--------|-------------------|
| Reassignment | Update assignee name in UI immediately after dropdown selection. If API call fails, revert to previous name and show error. |

## Accessibility

| Concern | Implementation |
|---------|----------------|
| Touch targets | All interactive elements use `min-h-12` (48px) per UX-02 requirement. Specialist group toggle buttons meet this minimum. |
| Form labels | "Assign To" dropdown has a visible `<label>` element with `htmlFor` binding |
| Select accessibility | Native `<select>` elements are inherently accessible -- screen readers announce options, keyboard navigation works natively |
| Specialist group toggle | Uses `<button>` element (not `<div onClick>`) for keyboard accessibility. Includes session count in label for context. |
| Assignee field tap-to-edit | Uses `title="Tap to reassign"` for sighted users. Tapping activates a native `<select>` which is screen-reader accessible. |
| Color contrast | All badge text colors meet WCAG AA against their backgrounds: blue-700 on blue-100, yellow-700 on yellow-100, orange-700 on orange-100, green-700 on green-100 all pass. |
| Loading states | Disabled elements use `disabled` attribute (not just visual styling) so screen readers announce correctly |
| Error messages | Error text uses `role="alert"` for screen reader announcement |
| Specialist view unchanged | No accessibility regressions for specialist users -- their view is identical to current implementation |
