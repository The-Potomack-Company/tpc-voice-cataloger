  <files>src/db/types.ts, src/db/index.ts, src/db/sessions.ts, src/hooks/useSessions.ts, src/hooks/useLongPress.ts, src/stores/uiStore.ts, src/tests/sessions.test.ts</files>
  <behavior>
    - Test: createSession returns numeric ID and session has status "active", notes "", correct name/mode
    - Test: updateSession changes name and bumps updatedAt
    - Test: softDeleteSession sets deletedAt to a Date, session excluded from active queries
    - Test: restoreSession clears deletedAt, session reappears in active queries
    - Test: permanentlyDeleteSession removes session and cascades to related items/audio/photos
    - Test: v2 migration sets status="active" and notes="" on existing records missing those fields
    - Test: useActiveSessions excludes soft-deleted and completed sessions (mock with Dexie in-memory)
  </behavior>
  <action>
    1. **Extend Session type** in `src/db/types.ts`: Add `status: "active" | "completed"`, `notes: string`, `deletedAt?: Date` fields to the Session interface.

    2. **Bump Dexie schema to v2** in `src/db/index.ts`: Add version(2) with updated sessions index `"++id, mode, status, updatedAt, createdAt, deletedAt"`. Re-declare ALL other tables unchanged. Add upgrade function that sets `status = "active"` and `notes = ""` on existing records. Keep version(1) declaration in place (Dexie requires it).

    3. **Create session CRUD module** `src/db/sessions.ts` with these exports:
       - `createSession(name: string, mode: "house" | "sale", notes?: string): Promise<number>` — adds session with status "active", returns ID
       - `updateSession(id: number, changes: Partial<Pick<Session, "name" | "notes" | "status">>): Promise<void>` — updates fields + bumps updatedAt
       - `softDeleteSession(id: number): Promise<void>` — sets deletedAt + bumps updatedAt
       - `restoreSession(id: number): Promise<void>` — clears deletedAt + bumps updatedAt
       - `permanentlyDeleteSession(id: number): Promise<void>` — deletes session + cascades to houseVisitItems, saleItems, audio, photos in a transaction
       - `getSessionById(id: number): Promise<Session | undefined>` — simple get

    4. **Create reactive hooks** `src/hooks/useSessions.ts`:
       - `useActiveSessions()` — useLiveQuery for status="active" AND no deletedAt, sorted by updatedAt desc
       - `useCompletedSessions()` — useLiveQuery for status="completed" AND no deletedAt, sorted by updatedAt desc
       - `useDeletedSessions()` — useLiveQuery for sessions WITH deletedAt, sorted by deletedAt desc
       - `useSession(id: number)` — useLiveQuery for single session by ID
       - `useSessionItemCount(sessionId: number)` — counts items (checks mode, queries appropriate table)

    5. **Create useLongPress hook** `src/hooks/useLongPress.ts`: Returns `{ onPointerDown, onPointerUp, onPointerLeave }` handlers. Takes callback and ms (default 500). Clears timer on up/leave.

    6. **Extend uiStore** `src/stores/uiStore.ts`: Add `recordingSessionId: number | null` and `setRecordingSession: (id: number | null) => void` to UIState. This persists to localStorage via Zustand persist, enabling interrupted recording detection on next app load.

    7. **Write tests** `src/tests/sessions.test.ts`: Use `fake-indexeddb` (or Dexie's built-in test support) for CRUD operations. Test create, update, soft delete, restore, permanent delete with cascade, and migration behavior.

    Anti-patterns to avoid:
    - Do NOT use `any` types — all Dexie operations use the Session interface
    - Do NOT forget to re-declare ALL tables in version(2) — Dexie drops undeclared tables
    - Default value for useLiveQuery must be defined outside component or be a stable reference (empty array literal `[]` is fine for lists, `0` for counts)
  </action>
  <verify>
    <automated>npx vitest run src/tests/sessions.test.ts --reporter=verbose</automated>
  </verify>
  <done>All CRUD operations work correctly; schema migration preserves data; soft delete filters work; reactive hooks compile; uiStore has recordingSessionId; all tests pass.</done>
</task>
