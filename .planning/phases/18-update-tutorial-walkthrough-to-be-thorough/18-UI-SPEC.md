### Progress Dot Colors

| State | Light | Dark |
|-------|-------|------|
| Active (current step) | `bg-accent` (#2563eb) | `bg-accent` (#2563eb) |
| Completed (past step) | `bg-accent` (#2563eb) | `bg-accent` (#2563eb) |

## Visual Hierarchy & Focal Point

### Walkthrough Step Layout

Each step occupies the full viewport, centered vertically and horizontally within the walkthrough container. Visual scanning hierarchy:

1. **Primary focal point: Step icon.** 64x64px Heroicon in accent color, centered at top of content area. Draws the eye first with its size and color contrast against the white/dark background.
2. **Secondary focal point: Step title.** 24px bold text immediately below the icon. Tells the user what this step is about.
3. **Tertiary focal point: Step description.** 16px normal-weight text below the title. Provides the explanation. Gray color (not black) to create visual hierarchy separation from the title.
4. **Action focal point: Navigation buttons.** Full-width accent-colored primary button at bottom of content area. "Back" text button to its left (from step 2 onward).

### Role Section Transition

When the walkthrough transitions from shared steps to role-specific steps, a role section label appears above the step icon:

```
"ADMIN FEATURES" or "YOUR WORKFLOW"
```

This 12px uppercase accent-colored label signals that the remaining steps are role-specific. It renders only on the first role-specific step, not on every subsequent role step.

### 3. Completion State Migration (Supabase)

**Database change:** Add `walkthrough_completed` boolean column to `profiles` table (default `false`).

**Completion flow:**
1. User taps "Start Cataloging" (final step) or "Skip tutorial" link.
2. Component calls `completeWalkthrough()` which:
   a. Updates Supabase `profiles` table: `UPDATE profiles SET walkthrough_completed = true WHERE id = {userId}`
   b. Updates local UI state to hide walkthrough immediately (optimistic).
3. On subsequent login on any device, walkthrough completion state is read from `profiles.walkthrough_completed`.

**Reset flow:**
1. User taps "Reset Walkthrough" in Settings.
2. Updates Supabase: `UPDATE profiles SET walkthrough_completed = false WHERE id = {userId}`
3. Updates local UI state. Next visit to Sessions page shows walkthrough.

**Local fallback:** Keep `hasCompletedWalkthrough` in `uiStore` as a cache for immediate UI responsiveness. On login, sync from Supabase `profiles.walkthrough_completed` value. This prevents a flash of walkthrough on page load while the Supabase query resolves.

### 4. Settings Page -- Reset Walkthrough Button (`Settings.tsx`)

No visual changes to the button itself. Same placement, same label.

### 5. Sessions Page Gate (`Sessions.tsx`)

**Modification to existing conditional.** The existing check at line 53 (`if (!hasCompletedWalkthrough)`) changes to read from the synced completion state. No visual change -- same gate, different data source.

## Interaction States

### Loading States

| Scenario | Visual Treatment |
|----------|-----------------|
| Walkthrough completion state loading | Sessions page renders normally (walkthrough hidden) until state resolves. If state resolves to `false`, walkthrough appears. This prevents a flash of walkthrough for returning users. Conservative default: assume completed. |
| Role detection for step filtering | Walkthrough renders shared steps immediately. Role-specific steps are appended once role resolves. If role detection is slow, the user progresses through shared steps (8 steps) which provides enough buffer. |
| "Start Cataloging" / "Skip" action in progress | Button shows `opacity-70` briefly during Supabase write. Optimistic: walkthrough dismisses immediately. |

### Error States

| Scenario | Visual Treatment |
|----------|-----------------|
| Supabase completion write fails | Walkthrough still dismisses (local state updated). Completion state will re-sync on next login. Worst case: user sees walkthrough again on next device. Console error logged. |
| Supabase completion read fails on login | Falls back to local `uiStore` cached value. If no cached value, defaults to `false` (show walkthrough). Console error logged. |
| Role detection fails | Walkthrough shows shared steps only (no role-specific steps appended). User still gets the core tutorial. Console error logged. |

### Transitions

| Transition | Animation |
|------------|-----------|
| Step forward (Next) | No animation. Instant content swap. Matches existing `Walkthrough.tsx` behavior. |
| Step backward (Back) | No animation. Instant content swap. |
| Walkthrough dismiss | Instant unmount. Matches existing behavior. |

Rationale: The existing walkthrough has no transitions. Adding animations is in Claude's discretion per CONTEXT.md, but the existing pattern of instant swaps is clean and avoids complexity. The progress dots provide sufficient orientation.

## Accessibility

| Concern | Implementation |
|---------|----------------|
| Touch targets | "Next", "Back", and "Start Cataloging" buttons use `min-h-12` (48px) per UX-02. "Skip tutorial" link has sufficient tap area via padding. |
| Keyboard navigation | "Next" and "Back" buttons are native `<button>` elements. Tab order: Back -> Next -> Skip. Enter/Space activates focused button. |
| Screen readers | Step title uses `<h2>` for heading hierarchy. Step counter provides context ("3 of 12"). Progress dots are decorative (`aria-hidden="true"`). |
| Color independence | Progress is communicated by both the step counter text ("3 / 12") and the filled progress dots. Not reliant on dot color alone. |
| Color contrast | Accent blue (#2563eb) on white (#ffffff) = 4.6:1 ratio (passes WCAG AA for large text/icons). Gray-600 (#4b5563) on white = 4.5:1 (passes AA). |
| Focus visibility | Buttons use default browser focus ring or Tailwind `focus-visible:ring-2 focus-visible:ring-accent` for keyboard users. |
| Motion | No animations -- instant step transitions. No motion-sensitivity concerns. |
