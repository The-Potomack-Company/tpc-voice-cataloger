# Phase 14 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Phase UI Surface

Phase 14 is primarily a data layer migration (Dexie to Supabase). The visible UI surface is intentionally small:

| New Component | Type | Description |
|---------------|------|-------------|
| MigrationSplash | Full-screen overlay | Blocking progress indicator during one-time data migration |
| Unsynced indicator | Inline badge | Subtle marker on items pending server sync (offline writes) |
| Offline banner | Status bar | Optional top banner on Sessions page when disconnected |

No new pages. No new navigation routes. No new forms. Existing pages (Sessions, SessionDetail, NewSession, ItemEntry) continue to render identically -- only their data source changes from Dexie to Supabase/Zustand.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding (`gap-1`, `p-1`) |
| sm | 8px | Compact element spacing (`gap-2`, `p-2`, `mt-2`) |
| md | 16px | Default element spacing (`p-4`, `mx-4`) |
| lg | 24px | Section padding (`p-6`) |
| xl | 32px | Layout gaps (`mb-8`) |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions:
- `py-0.5` (2px) on inline badge elements: Approved. The existing codebase uses `px-2 py-0.5 rounded-full` as the standard badge pattern across SessionCard, AccountRow, ItemCard, and Settings (8+ instances). Changing to `py-1` (4px) would be inconsistent with every other badge in the app. Phase 14's Unsynced Indicator badge inherits this established pattern.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (`text-base`) | 400 (normal) | 1.5 |
| Label | 14px (`text-sm`) | 400 (normal) | 1.43 |
| Heading | 24px (`text-2xl`) | 600 (`font-semibold`) | 1.33 |

Labels are differentiated from body text by size (14px vs 16px), not by weight. Only two weights are used in this phase's contract: 400 and 600.

Source: Login page uses `text-2xl font-semibold` for heading, `text-sm` for labels, default body text at 16px. Three sizes are sufficient for this phase's minimal UI surface.

Phase-specific typography:

| Element | Size | Weight | Notes |
|---------|------|--------|-------|

| Migration progress text | 14px (`text-sm`) | 400 (normal) | "12 of 45 items" counter |
| Unsynced badge text | 12px (`text-xs`) | 400 (normal) | Differentiated by size and color, not weight |
| Offline banner text | 14px (`text-sm`) | 400 (normal) | Matches OfflineIndicator pattern |

## Component Specifications

### MigrationSplash

Primary visual anchor is the `<h2>` heading -- all other elements (body text, progress bar, counter, buttons) support it.

```
Layout:
- Full-screen overlay, centered content (flexbox)
- Uses createPortal to document.body (same as ConfirmDialog)
- z-index: z-50 (same layer as ConfirmDialog)
- Not dismissible -- user must wait for completion

Structure:
  <div>  -- fixed inset-0, flex items-center justify-center, bg-white dark:bg-gray-900
    <div>  -- max-w-sm mx-4, text-center
      <h2>  -- text-2xl font-semibold, text-gray-900 dark:text-gray-100
      <p>   -- text-sm text-gray-500 dark:text-gray-400, mt-2
      <div> -- progress bar container, mt-8, w-full h-2 rounded-full bg-gray-200 dark:bg-gray-700
        <div> -- progress bar fill, h-full rounded-full bg-accent, width: {percent}%, transition-all duration-300
      <p>   -- text-sm text-gray-500, mt-3, "{current} of {total} items"

States:
  1. In progress: heading + body + animated progress bar + counter
  2. Complete: heading changes to "Migration complete", body changes, progress bar at 100%
  3. Error: heading changes, body shows skip count, two buttons (Retry + Skip and Continue)

Auto-dismiss: After success state shown for 1500ms, overlay fades out (opacity transition 300ms)
```

### Unsynced Indicator (Badge)

```
Placement: Inside ItemCard, after existing badges (mode badge, status badge)
Condition: Shown when item has a pending write-ahead queue entry

Structure:
  <span>  -- inline-flex items-center gap-1 text-xs px-2 py-0.5 rounded-full
            bg-amber-100 dark:bg-amber-900/30 text-amber-700 dark:text-amber-400
    <svg>   -- w-3 h-3, rotating refresh/sync icon (2-arrow circular)
    "Pending sync"

Spacing exception: py-0.5 (2px) used here to match the established badge pattern
across SessionCard, AccountRow, ItemCard, and Settings (see Spacing Scale exceptions).

Matches: Identical class pattern to "Recording interrupted" badge in SessionCard
Disappears: When write-ahead queue entry is processed and server confirms
```

### Offline Banner

```
Placement: Top of Sessions page, below navigation header, above session list
Condition: Shown when navigator.onLine is false

Structure:
  <div>  -- flex items-center justify-center gap-2 py-2 px-4
           bg-gray-100 dark:bg-gray-800
           role="status" aria-live="polite"
    <svg>   -- w-4 h-4 text-gray-500 dark:text-gray-400 (wifi-off icon, same as OfflineIndicator)
    <span>  -- text-sm text-gray-500 dark:text-gray-400
              "You're offline. Changes will sync when you reconnect."

Matches: Extends the existing OfflineIndicator pattern with explanatory text
```

## Interaction Contracts

### Migration Flow

1. User logs in (first time after Phase 14 deploy)
2. App detects Dexie has session/item metadata AND no ID mapping entries exist
3. MigrationSplash renders immediately, blocking all interaction
4. Progress bar advances as items migrate: `{current} of {total} items`
5. On success: splash shows "Migration complete" for 1500ms, then fades out
6. On partial failure: splash shows error state with Retry and Skip and Continue buttons
7. Retry: restarts migration for failed items only
8. Skip and Continue: dismisses splash; failed items remain in Dexie (user can retry from Settings later)
9. Migration never triggers again once ID mapping entries exist

### Offline Write Flow

1. User creates/edits a session or item while offline
2. Change applies immediately to Zustand store (optimistic)
3. "Pending sync" badge appears on the affected item
4. Entry added to write-ahead queue
5. When connectivity returns, queue processes in order
6. On success: badge disappears, no notification
7. On failure: badge remains, error logged to console; next reconnect retries

### Data Source Transition

All existing interactions (create session, edit item, delete session, export, etc.) work identically from the user's perspective. The only visible differences are:
- Route params change from numeric (`/session/5`) to UUID (`/session/abc-123-...`)
- "Pending sync" badge may appear briefly on items after create/edit
- Offline banner appears when disconnected
