<objective>
Fix UAT finding F2 (SC-2/SC-3). `AiFailureBanner` returns `null` whenever the integer `latestAudioId == null` (AiFailureBanner.tsx:39). For a cross-device / Dexie-cleared item the audio exists only in Supabase Storage, and `audioRecordsForItem` deliberately leaves Supabase-union rows with `id: undefined` (audioLookup.ts:54-65), so the `latestAudioId` reduce in both ItemList and ItemEntry yields `null`. Result: a failed item with recoverable server-side audio shows a red title and an INVISIBLE banner — no Retry path (GAP-5 in 42-RESEARCH.md).

Drive the banner's render + Retry off a **server-side audio existence boolean** (`hasServerAudio`) threaded from the parent — NOT the device-local integer (RESEARCH Open-Q3 recommendation, SHARED-3). The retry capability already exists: the gemini.ts orchestrator `processAudioWithAi(audioId, itemId, sessionId, isRetry)` re-claims `['failed','processing']` when `isRetry=true` and resolves the blob via `resolveAudioForAi` which falls through to Storage-by-`item_id` (no Dexie integer needed). The fix is *gating* + *plumbing a valid call*, not building a resolver.

LANDMINE (42-PATTERNS.md): two functions are named `processAudioWithAi`. The banner calls the **gemini.ts orchestrator** (`src/services/gemini.ts:202`), which internally calls the blob resolver imported as `resolveAudioForAi` (`src/services/processAudioWithAi.ts:21`). Wire the retry against the orchestrator — that is what already accepts `isRetry` and resolves by `item_id`.

Purpose: failed items with server-only audio get a working recovery path on both list + detail.
Output: hasServerAudio-gated banner + cross-device regression coverage; zero schema changes, zero new deps.
</objective>
