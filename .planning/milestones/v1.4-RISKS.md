# Milestone v1.4 Risks — Photo Notes

**Status:** PLANNED 2026-06-10. Each risk names its mitigation owner-phase.

## R-1 — Handwriting OCR accuracy (HIGH, core product risk)

Handwritten auction notes are messy: abbreviations, margin arrows, crossed-out lines,
mixed sketches and text. Gemini will misread some of it. Strategy is **containment,
not perfection**:

- **Per-field confidence** in the `responseSchema`; below threshold → field ships
  **blank + flagged**, never a guess (PHN-04). Starting threshold ~0.6, tuned from
  Phase 50 live-UAT findings — the threshold is a constant, not a schema concern.
- **Field-level fallback to blank-for-review**: a bad field costs the reviewer one
  inline edit; a confidently wrong field costs a bad catalog entry on RFC. Bias hard
  toward blank.
- **Raw OCR text preserved per draft** (transcript analog) so the reviewer can read
  what the model saw without re-fetching the photo.
- **Mandatory review gate** (locked decision 3) is the backstop: no OCR output reaches
  `items` without human confirmation. Extension import contract is protected by
  construction.
- Prompt encodes the house conventions (ALL-CAPS title shape, "the"-leading
  description, N×N×N in./cm. measurements) — reuse the `gemini.ts` SYSTEM_PROMPT
  formatting rules rather than re-deriving them.

Owner: Phase 47 (thresholds, prompt), Phase 48 (flag UX), Phase 50 (tuning).

## R-2 — Fields OCR cannot reliably produce (MEDIUM)

Per-field verdict against the items schema (`database.types.ts:339-424`):

| Field | OCR verdict | Handling |
|---|---|---|
| `title`, `description`, `category`, `condition` | Usually producible | Confidence-gated fill |
| `measurements` | Producible but format-fragile (must match N×N×N in./cm.) | Prompt-enforced format; blank on low confidence |
| `estimate` | Risky — digit misreads change meaning | Lower threshold tolerance; blank unless clearly legible |
| `receipt_number` (sale) | **Never trusted** — digit misread + DAT-8 unique constraint = wrong-item catalog entries | Pre-fill allowed, but explicit reviewer acknowledgment required before promotion (PHN-08); duplicate → named error (Phase 44 pattern) |
| `artist_*`, `medium` | Sometimes producible | Confidence-gated; no special handling |
| `transcript` | Repurposed: raw page OCR text | Always filled (review aid) |
| `mode`, `sort_order`, `ai_status`, timestamps | Not OCR's job | App-set: mode from session, sort_order appended, `ai_status='done'` on promote (keeps promoted items out of the audio-AI claim/queue logic) |

Owner: Phase 47 (schema/prompt), Phase 49 (contract doc + regression test).

## R-3 — Schema impact: draft storage (MEDIUM, migration required)

**Recommendation: new `item_drafts` table** (session-scoped RLS, columns mirroring
the catalog fields + `confidence` JSONB + `source_pages` + `raw_ocr_text` +
`status` `draft|promoted|discarded` + idempotency key), promoted via the existing
item-creation path.

Rejected alternative: a `draft` value on `items.ai_status` (or a new `items.status`).
Every existing consumer of `items` — list/counts, AI claim logic
(`ai_status` check constraint `pending|processing|done|failed|queued`), export,
extension import, optimistic-locking paths — would need a draft-exclusion clause;
one missed spot leaks OCR garbage into the export contract. A separate table makes
drafts invisible by construction.

Consequences: one new migration (Phase 47 wave 0) + `npm run db:types` regen +
cross-app schema ceremony (`_workspace/Schema/schema.md` + `migrations.md` first —
schema is shared; D-046: Claude owns schema plumbing). Dashboard/extension are
unaffected (new table, no changes to theirs).

## R-4 — Note-page durability scope (LOW/MEDIUM, scope decision for Phase 47 discuss)

Phase 46 keeps pages Dexie-only (offline capture, zero migration). Phase 47 must
decide page persistence:

- **Option A (recommended): persist pages** to the existing `photos` bucket under a
  `notes/{sessionId}/` prefix + small `note_pages` table — cross-device review,
  audit trail, consistent with the v1.3 durable-audio lesson (Phase 32).
- **Option B: ephemeral** — pages live in Dexie, base64 straight into the process
  call, discarded after promotion. Smaller scope, no migration row, but review is
  device-locked and there is no audit trail.

Decide at `/gsd-discuss-phase 47`. Option A adds the `note_pages` table to the same
wave-0 migration as `item_drafts`.

## R-5 — Gemini image limits & cost at usage level (LOW for cost, MEDIUM for limits)

- **Request size:** proxy/contract body cap is 25 MB (Cloud Run accepts 32 MB).
  Pages resized to maxDimension ~2048 JPEG (~0.4-0.8 MB each, ×1.33 base64) →
  comfortable up to ~12-20 pages per request. UI caps the batch and tells the user
  to split rather than silently truncating (Phase 47 SC-3).
- **Cross-batch segmentation:** an item spanning a batch boundary gets split into
  two drafts. Mitigation: generous batch cap + reviewer merges via edit/discard.
  Accepted for v1.4; revisit only if UAT shows it is common.
- **Cost:** image inputs on `gemini-2.5-flash` are cheap relative to the existing
  audio traffic (~hundreds of tokens per image tile set); a 100-item house visit is
  on the order of a dozen calls — orders of magnitude below continuous mode's
  ~480 calls/2h that D-050 killed. Phase 50 measures real usage. **Exact current
  per-image token pricing was not verified during planning — confirm at Phase 47
  planning; treat any quota/plan change it forces as a BILLING item.**

## R-6 — tpc-ai-proxy payload compatibility (MEDIUM, verify before Phase 47)

The cataloger's proxy contract today carries audio `inlineData`. If the proxy
validates/filters payload shape (mime types, part counts) or pins request schemas,
multi-image payloads may be rejected. Verification task at Phase 47 planning:
inspect the tpc-ai-proxy request handling (separate repo, Cloud Run, deploy via
hardened `deploy.yml` — Phase 40.1).

- If a proxy change is needed: it is a **deploy** of an existing service (allowed),
  not a new billed resource. Coordinate as a cross-app touch (D-056 contract,
  `ALLOWED_ORIGINS` untouched, JWT verify preserved).
- If anything requires enabling a new API/model tier on the key: **BILLING item,
  stop and surface for user triage.**

## R-7 — Review-queue / concurrency interactions (LOW)

Drafts are single-writer (the processing call creates; the reviewer edits) so the
Phase 39 lost-write class mostly doesn't apply — but **promotion** writes through
the existing item-creation + field-write paths, which already carry
`preconditionUpdate`. Keep promotion on those paths (Phase 48 SC-2); do not write
`items` directly from draft code. Idempotency key on the process action (PHN-09)
prevents duplicate drafts from retries.
