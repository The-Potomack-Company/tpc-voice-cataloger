<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Supabase UPDATE | The `updated_at` precondition snapshot comes from the client's own prior read. A forged value either matches legitimately (harmless) or 0-rows (safe). RLS still gates row access. |
| reconcile loop ↔ Postgres | The loop re-reads after every conflict; a re-read denied by RLS returns nothing, identical in shape to a precondition miss. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-01 | Tampering | precondition write in `optimisticUpdate.ts` | mitigate | `.eq("updated_at", prev).select()` + `data.length === 0` conflict detection; a stale write 0-rows and reconciles instead of clobbering. Test `optimistic-update.test.ts` proves 0-row ≠ success. |
| T-39-02 | Denial of Service | reconcile loop (RLS-deny masquerading as conflict → infinite loop) | mitigate | 3-attempt cap (D-07) bounds the loop. A re-read returning NOTHING ends the loop with `deleted: true` (does not retry) — this disambiguates "row exists, token moved" (retry) from "RLS-deny / deleted" (stop, surface). Highest-value threat this phase; covered by the re-read-returns-nothing test case. |
| T-39-03 | Tampering | client forging a fake `updated_at` | accept | Token is from the client's own prior read; a forged value either matches (legitimate) or 0-rows (safe). RLS gates row access regardless. Low risk per RESEARCH Security table. |
| T-39-SC | Tampering | npm/pip/cargo installs | mitigate | Phase adds NO packages; no install task. No `[ASSUMED]`/`[SUS]` package to gate. |
