<objective>
Fix the confirmed house/sale `++id` keyspace collision in `src/services/photoMigration.ts`. The `migrateExistingPhotos` loop resolves a photo's Dexie `itemId` to its Supabase UUID with an inline `db.idMapping.where("oldId").equals().filter(m => m.type === "item")` query that ignores whether the photo came from a house item or a sale item. Because `houseVisitItems` and `saleItems` have INDEPENDENT `++id` keyspaces, a sale photo can resolve to a house item's UUID when the two share the same legacy integer id — silent cross-table photo misattribution (UAT finding 38-3).

The fix: replace the inline query with the existing, already-correct helper `getNewIdByOldId(photo.itemId, "item", photo.itemType)` from `src/db/idMapping.ts`, which applies the `itemTable` discriminator the same way `src/db/migration.ts:migrateToSupabase` already does.

Purpose: Stop sale-item photos from being uploaded against the wrong (house) Supabase item, and vice versa.
Output: A scoped lookup in photoMigration.ts plus a regression test that fails on the old code and passes on the fixed code.
</objective>
