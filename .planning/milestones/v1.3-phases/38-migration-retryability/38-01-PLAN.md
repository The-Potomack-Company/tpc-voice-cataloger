<objective>
Make the Dexie→Supabase migration idempotent and correctly partial-aware at the data layer: a per-row `needsMigration()` predicate (D-01/D-02), an `[oldId+type]` reverse index + `getNewIdByOldId` helper (D-03/D-04), lookup-before-insert guards on the session insert and both item loops (D-05), a `failed` vs `alreadyMigrated` counter split with `partial = failed > 0` (D-10), and an `exportHistory` cleanup gated on a post-run ground-truth `needsMigration()` call (D-09).

Purpose: A partial migration (DAT-1 / PR #24 preserves the failed recovery set in Dexie) is today reported "complete" by the count-based `needsMigration()`, and a naive retry would duplicate every preserved Supabase row because there is no oldId→newId reverse lookup. This plan closes both — SC1 and SC2.

Output: Modified `src/db/{index.ts,idMapping.ts,migration.ts}`; a flipped existing needsMigration test + a new idempotency/retry-after-partial test file. UI surfacing (banner) is Plan 02.
</objective>
