<objective>
REL-2: make duplicate Gemini spend structurally impossible across tabs/processes/devices via a DB-atomic claim (D-01). Before any tab processes a queued item it issues a conditional update — `update items set ai_status='processing', claimed_at=now() where id=? and ai_status='queued'` chained with `.select("id")` — and only the tab that gets a row back proceeds. A stale-claim reclaim pass (D-02) re-queues items stuck in `processing` past STALE_MS (~5min) so a dead tab cannot strand work. No BroadcastChannel (D-03).

Purpose: today `CONCURRENCY=4` plus zero cross-tab coordination means concurrent drains across tabs re-process the same item, burning duplicate Gemini calls and racing updates. The per-tab `draining` boolean only dedupes within one tab.

Output: claim + reclaim wired into `offlineQueue.ts` + extended tests proving exactly-once across 4 concurrent drains.

Sequencing: depends_on 33-01 (REL-1) which owns the backoff/attempt changes in the SAME file. Both are Wave 1 but run sequentially — they MUST NOT execute in parallel (file conflict on offlineQueue.ts). This plan layers the claim onto 33-01's hardened drain.
</objective>
