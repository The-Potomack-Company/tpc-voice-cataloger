<objective>
REL-1: replace the `MAX_RETRIES=2` immediate-retry loop in `offlineQueue.ts` with persisted-attempt backoff. The drain reads `claimed_at`+`ai_attempts` from the queue select, skips items still inside their full-jitter backoff window (killing the retry storm on every `online` flip), increments `ai_attempts` on failure, and marks the item `ai_status='failed'` once the attempt cap (5) is exceeded. Folds in the #17 net-abort-requeue follow-up by treating offline/network errors as transient via the shared classifier.

Purpose: today every `online` event re-drains the whole queue and immediately retries failures twice with no persistence, so a permanently-failing item burns Gemini calls forever. Persisted attempts + a backoff window make retries cross-tab/device consistent and bounded.

Output: hardened `offlineQueue.ts` drain + extended offline-queue tests.

Sequencing: this plan OWNS the offlineQueue.ts changes for REL-1. Plan 33-02 (REL-2) layers the DB-atomic claim onto the same file and `depends_on` this plan — they share Wave 1 but run sequentially to avoid a merge conflict (per wave_guidance).
</objective>
