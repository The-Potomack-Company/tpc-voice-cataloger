# Phase 40.1: Harden tpc-ai-proxy deploy.yml CI before proxy push - Context

**Gathered:** 2026-06-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Harden the `tpc-ai-proxy` Cloud Run deploy workflow (`.github/workflows/deploy.yml`,
**lives in the sibling `../tpc-ai-proxy` repo**) so the proxy cannot ship without
the secrets its fail-closed config now requires. Phase 40 added net-new
fail-fast throws in `tpc-ai-proxy/src/config.js` for `SUPABASE_URL` and
`SUPABASE_ANON_KEY` (Supabase-JWT bearer verify); the current `deploy.yml` does
not set them, so the next push to proxy `main` crash-loops prod. This phase wires
the missing env, adds a CI preflight gate so a future missing secret fails the run
instead of prod, corrects the cataloger origin allowlist, and extends the same
hardening to the dev service.

**Out of scope:** the proxy's application/auth code (already done in Phase 40 —
Claude-owned, Codex barred per D-046); changing the WIF/keyless auth mechanism;
GCP/Secret Manager provisioning beyond setting GH repo secrets.

</domain>

<decisions>
## Implementation Decisions

### Guardrail strength
- **D-01:** Preflight gate. Add `SUPABASE_URL` + `SUPABASE_ANON_KEY` to the
  deploy `env_vars`, AND add a CI step that runs **before** the deploy step and
  asserts every required GH secret (`SUPABASE_URL`, `SUPABASE_ANON_KEY`,
  `ALLOWED_ORIGINS{_PROD,_DEV}`, `GCP_WIF_PROVIDER`, `GCP_DEPLOY_SA`) is
  non-empty — fail fast in CI with a clear message instead of crash-looping
  Cloud Run. (Post-deploy health-check curl was considered and deferred — not
  required for this landmine close.)

### Secret sourcing (Supabase)
- **D-02:** Both `SUPABASE_URL` and `SUPABASE_ANON_KEY` become GH **repo
  secrets**, referenced via `${{ secrets.* }}` in `env_vars`. No values in git,
  even though the URL is public and the anon key is publishable — consistency
  over micro-optimizing.

### ALLOWED_ORIGINS correction
- **D-03:** The current `ALLOWED_ORIGINS` GH secret is extension-only and would
  drop the cataloger origin. Update it via `gh secret set` (CLI, in scope) so the
  deployed proxy keeps `https://tpc-cataloging-app.vercel.app` allowlisted.
  Because dev needs a *different* value (dev extension id), this splits into
  **per-service secrets** — see D-05.

### Dead env var fix
- **D-04:** Rename `BODY_CAP_BYTES` → `MAX_BODY_BYTES` in `env_vars`. config.js
  reads `MAX_BODY_BYTES`; the current `BODY_CAP_BYTES=26214400` is dead and the
  25 MB cap silently falls back to the code default. Small correctness fix, folded
  because we're already editing the `env_vars` block.

### Dev service coverage
- **D-05:** Wire `tpc-ai-proxy-dev` into the same workflow via a **matrix** so
  every push to `main` deploys **both** prod and dev in lockstep. Each service
  uses its own origin secret (`ALLOWED_ORIGINS_PROD` / `ALLOWED_ORIGINS_DEV`) and
  shares the Supabase + GCP secrets. The single shared `ALLOWED_ORIGINS` secret is
  therefore retired in favor of the two per-service secrets.
  - Live values to seed the secrets (from project memory `tpc-ai-proxy-deploy`):
    - prod: `chrome-extension://plielbnfpgcjjohnehcnagmlkmhjcpdn,https://tpc-cataloging-app.vercel.app`
    - dev:  `chrome-extension://nhpailgcmoimhpgcaaknmglchlkplbig,https://tpc-cataloging-app.vercel.app`

### Claude's Discretion
- Exact matrix shape (GHA `strategy.matrix` vs two jobs), preflight step
  implementation (inline `if -z` bash vs a tiny composite), and where the
  per-service `service` / `project_id` / `region` values are parameterized —
  planner/executor decide. `ALLOWED_MODELS` stays manually-set + revision-preserved
  (comma-in-value breaks deploy-cloudrun env_vars parsing) — do NOT add it to
  `env_vars`.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `deploy-cloudrun@v2` action already wired with `secrets:` (Secret Manager refs)
  and `env_vars:` blocks — extend, don't rebuild.
- Keyless WIF auth (`google-github-actions/auth@v2` with `GCP_WIF_PROVIDER` +
  `GCP_DEPLOY_SA`) already works — leave untouched.

### Established Patterns
- Fail-fast/fail-closed is the proxy's house style (config.js throws on missing
  required env). The CI preflight gate (D-01) mirrors that pattern at the deploy
  layer.
- Comma-bearing values (`ALLOWED_ORIGINS`, `ALLOWED_MODELS`) are the known
  footgun: deploy-cloudrun@v2 `env_vars` splits VALUES on commas — even when the
  value arrives via a single `${{ secrets.X }}` reference (RESEARCH.md corrects an
  earlier CONTEXT assumption; git history `d8fced8`→`51ba8e8` proves it). The
  current `ALLOWED_ORIGINS` secret only survives because its value is comma-free
  (extension-only). The D-05 per-service values DO contain a comma, so the
  `env_vars` entry MUST quote the reference
  (`ALLOWED_ORIGINS="${{ secrets.ALLOWED_ORIGINS_PROD }}"`) or the cataloger origin
  is silently dropped. Also set `env_vars_update_strategy: merge` to preserve the
  manually-set `ALLOWED_MODELS` on the service. `ALLOWED_MODELS` stays out of
  `env_vars` entirely. See 40.1-RESEARCH.md for the canonical pattern.

### Integration Points
- GH repo secrets on `The-Potomack-Company/tpc-ai-proxy` (org, NOT `spoods/`) —
  existing: `ALLOWED_ORIGINS`, `GCP_DEPLOY_SA`, `GCP_WIF_PROVIDER`. Create:
  `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `ALLOWED_ORIGINS_PROD`,
  `ALLOWED_ORIGINS_DEV`; retire `ALLOWED_ORIGINS` after cutover. All four new ones
  settable via `gh secret set` (in scope).
- Cloud Run services `tpc-ai-proxy-prod` + `tpc-ai-proxy-dev`, project
  `gen-lang-client-0662587427`, region `us-east1`.
