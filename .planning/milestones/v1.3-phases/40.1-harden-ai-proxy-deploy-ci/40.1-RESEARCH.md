## Summary

This is a **config-only CI hardening phase** in the sibling repo `../tpc-ai-proxy` (GitHub: `The-Potomack-Company/tpc-ai-proxy`). The single file to edit is `.github/workflows/deploy.yml`. No application code changes, no new packages. The work is: (1) add a preflight secret-presence gate, (2) add Supabase env vars, (3) fix a dead env var name, (4) correct the origin allowlist via per-service secrets, and (5) fan the deploy out to both prod and dev via a matrix. The ship order is hard-constrained: **harden the workflow and create the secrets BEFORE pushing the two unpushed phase-40 proxy commits** (`18ef932`, `f52adc4`), which add fail-closed `SUPABASE_*` throws that the current workflow does not satisfy.

The single highest-risk item — and the one place the CONTEXT's stated assumption is **partially wrong** — is comma-safety. CONTEXT states "ALLOWED_ORIGINS works today only because it comes through a single secret reference." Git history proves otherwise: `deploy-cloudrun`'s `env_vars` parser splits **values** on commas, and the current `ALLOWED_ORIGINS` secret works today **only because its value is a single comma-free extension origin** (set 2026-05-29 as extension-only). The new per-service values D-05 seeds **contain a comma** (`chrome-extension://...,https://...`). Passing a comma-bearing value through `${{ secrets.X }}` into `env_vars` will be split into two malformed `KEY=VALUE` pairs unless the value is **quoted** or the comma is **backslash-escaped**. This must be handled or both deploys break on the comma.

**Primary recommendation:** Use a `strategy.matrix` over `[prod, dev]`, map per-leg secret names through `matrix.include`, add a bash preflight step that maps required secrets to env then tests `-z` (GitHub forbids `secrets.*` in `if:`), set `env_vars_update_strategy: merge` to preserve the manually-set `ALLOWED_MODELS`, and **quote the comma-bearing `ALLOWED_ORIGINS` value** in the `env_vars` block: `ALLOWED_ORIGINS="${{ secrets.ALLOWED_ORIGINS_PROD }}"`.

### Pattern 1: Preflight secret-presence gate (D-01)
**What:** A bash step that fails the run if any required secret is empty, BEFORE auth/deploy.
**When to use:** Always run before the deploy step. **GitHub does NOT allow `secrets.*` in `if:` conditions** — you cannot write `if: ${{ secrets.SUPABASE_URL == '' }}`. The canonical pattern is: map each secret into the step's `env:`, then test `-z` in `run:`.
**Example:**
```yaml
# Source: GitHub Actions docs — secrets are only available via the secrets context
# in `env:`/`with:`, never in `if:`. https://docs.github.com/actions/security-guides/using-secrets-in-github-actions
- name: Preflight — assert required secrets are set
  env:
    SUPABASE_URL: ${{ secrets.SUPABASE_URL }}
    SUPABASE_ANON_KEY: ${{ secrets.SUPABASE_ANON_KEY }}
    ALLOWED_ORIGINS: ${{ matrix.allowed_origins }}   # already mapped per-leg, see Pattern 2
    GCP_WIF_PROVIDER: ${{ secrets.GCP_WIF_PROVIDER }}
    GCP_DEPLOY_SA: ${{ secrets.GCP_DEPLOY_SA }}
  run: |
    missing=()
    for v in SUPABASE_URL SUPABASE_ANON_KEY ALLOWED_ORIGINS GCP_WIF_PROVIDER GCP_DEPLOY_SA; do
      if [ -z "${!v}" ]; then missing+=("$v"); fi
    done
    if [ ${#missing[@]} -ne 0 ]; then
      echo "::error::Missing required secret(s): ${missing[*]}. Set them on the repo before deploy." >&2
      exit 1
    fi
    echo "Preflight OK — all required secrets present."
```
- `"${!v}"` is bash indirect expansion — reads the value of the variable whose name is in `$v`.
- `::error::` emits a GitHub annotation so the failure is visible in the run summary.
- **Per-leg vs once:** Running it inside the matrix job (per-leg) is simplest and the CONTEXT explicitly allows it. The only per-leg-divergent secret is `ALLOWED_ORIGINS` (prod vs dev); the shared secrets are checked redundantly on both legs, which is harmless. A separate single `needs:`-gate job is also valid (Claude's discretion) but adds a job and the `needs:` wiring; prefer the inline per-leg step unless the planner wants a hard gate that blocks both legs on one check.

### Pattern 2: Matrix over services with per-leg secret selection (D-05)
**What:** One job, `strategy.matrix` with `include` carrying the per-service `service` name and the per-service origin value. Secrets cannot be indexed dynamically by a matrix variable (e.g. `secrets[matrix.x]` is not supported), so map each leg's origin secret explicitly in `matrix.include`.
**When to use:** Two near-identical deploys differing only by `service` + `ALLOWED_ORIGINS`.
**Example:**
```yaml
# Source: GitHub Actions matrix docs https://docs.github.com/actions/using-jobs/using-a-build-matrix-for-your-jobs
jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write
    strategy:
      fail-fast: false   # let dev finish even if prod fails, and vice-versa
      matrix:
        include:
          - service: tpc-ai-proxy-prod
            allowed_origins: ${{ secrets.ALLOWED_ORIGINS_PROD }}
          - service: tpc-ai-proxy-dev
            allowed_origins: ${{ secrets.ALLOWED_ORIGINS_DEV }}
    steps:
      - uses: actions/checkout@v4
      # ... preflight (uses matrix.allowed_origins) ...
      # ... auth@v2 / setup-gcloud@v2 (shared secrets) ...
      - name: Deploy to Cloud Run
        uses: google-github-actions/deploy-cloudrun@v2
        with:
          service: ${{ matrix.service }}
          region: us-east1
          source: .
          project_id: gen-lang-client-0662587427
          env_vars_update_strategy: merge   # preserves manually-set ALLOWED_MODELS across revisions
          secrets: |
            GEMINI_API_KEY=gemini-api-key:latest
          env_vars: |
            ALLOWED_ORIGINS=${{ matrix.allowed_origins }}
            SUPABASE_URL=${{ secrets.SUPABASE_URL }}
            SUPABASE_ANON_KEY=${{ secrets.SUPABASE_ANON_KEY }}
            RATE_LIMIT_RPM=15
            MAX_BODY_BYTES=26214400
          flags: "--max-instances=1 --allow-unauthenticated"
```
> **NOTE:** the `ALLOWED_ORIGINS` line above is shown UNQUOTED for structural clarity — **it must be quoted in the real workflow** (Pattern 3 / Pitfall 1). `fail-fast: false` is recommended so a prod failure doesn't cancel an in-flight dev deploy.

### Pattern 3: Comma-safe env_vars value (the critical one)
**What:** Quote the comma-bearing value so the action's `env_vars` parser does not split it into two KEY=VALUE pairs.
**Why:** `deploy-cloudrun`'s `env_vars` is "comma-separated **or** newline-separated `KEY=VALUE`. Keys or values that contain separators must be escaped with a backslash (e.g. `\,`) **unless quoted**." `[CITED: github.com/google-github-actions/deploy-cloudrun README]`
**Example (recommended — quote the whole value):**
```yaml
env_vars: |
  ALLOWED_ORIGINS="${{ matrix.allowed_origins }}"
```
Alternative (escape the comma, fragile — requires baking `\,` into the secret value): not recommended.

### Anti-Patterns to Avoid
- **`secrets.*` inside `if:`** — silently always-false / context error. GitHub only exposes `secrets` in `env:`/`with:`. Map to env first.
- **Putting a literal comma value directly in `env_vars`** — exactly what broke `ALLOWED_MODELS` (commit `51ba8e8`). Never inline `ALLOWED_MODELS=a,b`. Keep it out of `env_vars` entirely (Claude's discretion locks this).
- **`env_vars_update_strategy: overwrite`** — would wipe the manually-set `ALLOWED_MODELS` on every deploy. Use `merge` (the default, but set it explicitly for intent).
- **`secrets[matrix.key]` dynamic indexing** — unsupported; map per-leg secrets via `matrix.include`.
- **gcloud `^|^` caret delimiter in the action** — that's a `gcloud run deploy` CLI codepath, NOT the action's `env_vars` parser. Commit `d8fced8` already removed the caret hack when moving to the action's multiline syntax. Do not reintroduce it; quoting is the action-layer fix.

## Common Pitfalls

### Pitfall 1: Comma in ALLOWED_ORIGINS splits the env_vars value
**What goes wrong:** `ALLOWED_ORIGINS=${{ secrets.ALLOWED_ORIGINS_PROD }}` expands to `ALLOWED_ORIGINS=chrome-extension://...,https://tpc-cataloging-app.vercel.app`. The action splits on the comma → second pair `https://tpc-cataloging-app.vercel.app` has no `=` → either parse error or a dropped/garbage env var. The cataloger origin is exactly the one silently lost — recreating the original bug D-03 set out to fix.
**Why it happens:** `env_vars` treats comma as a pair separator. The CONTEXT's claim that "single secret reference" makes it safe is **wrong** — the current secret is safe only because its value has no comma. `[VERIFIED: git show d8fced8 + commit 51ba8e8 — literal-comma ALLOWED_MODELS was removed for exactly this reason]`
**How to avoid:** Quote the value: `ALLOWED_ORIGINS="${{ matrix.allowed_origins }}"`.
**Warning signs:** Proxy boots but rejects the cataloger origin with CORS; or deploy step errors `Invalid env var spec`.

### Pitfall 2: `secrets.*` in `if:` is always evaluated empty
**What goes wrong:** A preflight written as `if: ${{ secrets.SUPABASE_URL == '' }}` never fires (or errors), giving false confidence.
**Why it happens:** GitHub deliberately excludes the `secrets` context from `if:` evaluation. `[CITED: docs.github.com/actions secrets context availability]`
**How to avoid:** Map secret → step `env:`, test with `-z` in `run:` (Pattern 1).
**Warning signs:** Preflight "passes" even when a secret is unset.

### Pitfall 3: `overwrite` strategy nukes ALLOWED_MODELS
**What goes wrong:** Switching to `env_vars_update_strategy: overwrite` (or a `--set-env-vars` equivalent) deletes the manually-set `ALLOWED_MODELS` (which lives only on the service), so the proxy falls back to `DEFAULT_MODELS` in config.js — a silent behavior change.
**How to avoid:** Use `merge` (default; set explicitly). Memory note already says "don't use `--set-env-vars`" — same principle at the action layer.
**Warning signs:** Allowed model set shrinks to the code default after a deploy.

### Pitfall 4: Pushing before secrets exist
**What goes wrong:** If the proxy is pushed before `SUPABASE_URL`/`SUPABASE_ANON_KEY` secrets exist, the new preflight correctly **fails the CI run** (good — no crash-loop), but if the preflight were skipped, config.js would throw at boot and crash-loop prod.
**How to avoid:** Strict ship order (see §7). Create secrets first.
**Warning signs:** Run fails at preflight with the missing-secret annotation (this is the **designed** safe failure, not a bug).

### Pitfall 5: deploy-cloudrun multiline `secrets:`/`env_vars:` newline parsing (historical)
**What goes wrong:** Older issues (#303) reported newlines leaking into gcloud args with the `|` block syntax.
**Why it happens:** Early parser bug; the workflow already uses the multiline block successfully today (commit `d8fced8`), so v2.7.6 handles it. Documented here only so the executor doesn't "fix" working syntax.
**How to avoid:** Keep the existing multiline block; just add lines. Don't switch to inline comma-joined form.

## Verification Approach (§6 — how to validate without crash-looping prod)

Ranked safest → most involved:

1. **Static YAML + actionlint (no network).** `yamllint`/Python `yaml.safe_load` proves the file parses; `actionlint` (if installed) catches `secrets`-in-`if`, matrix typos, undefined contexts. Runnable by the executor now. **Recommended baseline.**
2. **Dry verify secrets exist before push.** `gh secret list --repo The-Potomack-Company/tpc-ai-proxy` — confirm all five required secrets present. This is exactly what the preflight asserts; checking it manually first means the first real run won't fail at preflight.
3. **`workflow_dispatch` on the hardened workflow (already enabled in deploy.yml `on:`).** After merging the hardened workflow to the proxy `main` BUT before the risky source push — actually the workflow change *is* a push to main, so this collapses into the real deploy. To get a no-op dry run, the cleanest is the preflight: if secrets are missing the run stops before any GCP mutation. There is no built-in `--dry-run` for deploy-cloudrun.
4. **Manual `gcloud run deploy` to `-dev` only** as a smoke (user-only — needs reauth). Validates the env set against the live config.js throws on a non-prod service first. Use `--update-env-vars` (never `--set-env-vars`, per memory) with `^|^` caret delimiter for the comma value (CLI codepath — different from the action).
5. **Real push** — gated behind 1–4. The preflight is the in-CI safety net: a missing secret fails fast instead of crash-looping.

The executor can run 1 and 2 directly. 3/5 are the actual deploy. 4 needs the user (gcloud reauth).
