# Phase 36 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This is an error-visibility/transactional-correctness polish phase. It introduces **no new visual language** — it routes currently-silent failure paths through the existing v1.2 design system (DAT-4 `notificationStore` → `ErrorToast`, plus the `WarnBanner` primitive for partial-state copy). All tokens, primitives, and themes below are **inherited from v1.2 (SHIPPED 2026-05-13)**, not declared fresh. The contract's job here is to lock the *failure-feedback states, copy, and retry affordances*, not to re-spec the design system.

## Color

Inherited semantic tokens from `tpc-unified-tokens.css` (light + dark). Failure surfaces draw **only** from the `err` / `warn` families; success/partial-recovery copy may use `ok`. Accent is NOT used for error feedback.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--bg` (`oklch(0.985 0.003 240)` light / `oklch(0.19 0.015 255)` dark) | App background, surfaces |
| Secondary (30%) | `--bg-2` / `--bg-3` | Cards, sidebar, nav |
| Accent (10%) | `--accent` (`oklch(0.58 0.13 225)`) | NOT used for error feedback this phase |
| Destructive / error | `--err` (`oklch(0.55 0.13 28)`), `--err-wash` | Error toast surface, error state copy |
| Warning / partial | `--warn` (`oklch(0.62 0.11 75)`), `--warn-wash` | Partial-state banners (migration partial, partial-recovery) |
| OK / recovered | `--ok` (`oklch(0.55 0.10 155)`), `--ok-wash` | Honest success copy after a clean operation |

Accent reserved for: primary CTAs, focus rings, live-recording emphasis (v1.2 contract). **Error feedback never uses accent** — error = `err` family, partial = `warn` family. This keeps the 10% accent budget intact and gives failures a distinct, learnable color signal.

Error toast surface uses the existing `ErrorToast` treatment: solid `bg-red-700 dark:bg-red-800` + white text (intentionally high-salience, pre-existing DAT-4 styling — retained, not restyled). Partial-state banners use the lower-salience `WarnBanner` (`bg-warn-wash text-warn`).

## Interaction Contract — Failure Feedback States

### Toast lifecycle (DAT-4, modified per CONTEXT decisions)

| Behavior | Rule | Source |
|----------|------|--------|
| Concurrency | Single-slot, latest-message-wins | D-04 |
| Dedupe | Identical-to-current message does not re-fire (no flicker/spam) | D-05 |
| Retryable = sticky | When a `retry` callback is attached, drop the 6s auto-dismiss; toast persists until user taps Retry or Dismiss | D-06 |
| Informational = auto-dismiss | No retry callback → keep existing 6s auto-dismiss | D-06 |
| Retry semantics | Retry re-runs the whole operation (export idempotent; fetch refetches). No partial/targeted retry | D-08 |
| Affordances | Retry (semibold underline) + Dismiss (`x`); both already in `ErrorToast` | existing |

### Central copy mapping — `toUserMessage(err)` (D-09)

All touchpoints (login, fetch, admin load, export) funnel through one helper. Raw Supabase/JSON error text must **never** reach the user.

| Error class | User-facing copy | Codex ref |
|-------------|------------------|-----------|
| Supabase auth failure (bad credentials) | `Wrong email or password` | #21 |
| Network / fetch failure | `Connection problem — try again` | #27, #28 |
| Unknown / unmapped | `Something went wrong` | fallback |

## Copywriting Contract

Per-touchpoint failure copy. Wording within the spirit of D-09 is Claude's discretion; the strings below are the prescribed defaults the executor implements unless the planner overrides.

| Touchpoint | State | Copy | Affordance |
|------------|-------|------|------------|
| Export (Codex #9, #10) | failure | `Export failed — your data wasn't downloaded.` | Retry (sticky) |
| New session / import (Codex #7, #8) | failure after client-side rollback (D-01) | `Couldn't create the session — nothing was saved. Try again.` | Retry (sticky) |
| Import (partial guard) | rollback fired | `Import didn't finish — changes were undone. Try again.` | Retry (sticky) |
| Silent fetch (Codex #27, #28) | failure | `Connection problem — try again` | Retry (sticky) |
| Admin role/account load (Codex #16–20) | failure | `Couldn't load accounts — try again.` | Retry (sticky) |
| Login (Codex #21) | bad credentials | `Wrong email or password` | none (form re-entry) |
| Login | network | `Connection problem — try again` | none (form re-entry) |
| Migration banner (Codex #2, D-07) | `partial` flag set | `Some items couldn't be migrated. Your data is safe.` (warn banner — must NOT claim full success) | none in P36 (Phase 38 adds Retry) |
| Migration banner | clean success | `Your data is up to date.` (or existing success copy) | none |

### Required template fields

| Element | Copy |
|---------|------|
| Primary CTA | `Retry` (re-run operation; sticky on retryable toasts) |
| Empty state heading | N/A — no new empty states introduced this phase |
| Empty state body | N/A |
| Error state | `{operation} failed — {what happened to your data}. Try again.` — every silent path produces a visible toast or `WarnBanner`; no console-only failures (ROADMAP test gate) |
| Destructive confirmation | N/A — no new destructive actions. Existing delete confirmations unchanged. Client-side rollback (D-01) runs compensating deletes automatically and is NOT user-confirmed (it's recovery, not a user-initiated destructive action) |

**Copy principles (enforce in review):**
- State what happened to the user's data ("nothing was saved", "your data is safe", "changes were undone") — reassurance is the point of this phase.
- Use em-dash + lowercase continuation for the action hint (`— try again.`).
- Never surface raw Supabase/JSON/stack text (D-09).

## Accessibility (inherited, must be preserved)

- `ErrorToast` carries `role="alert"` + `aria-live="assertive"` — keep; new errors are announced to screen readers.
- `WarnBanner` carries `role="status"` (polite) — correct for partial-state banners (not interruptive).
- Retry/Dismiss buttons keep visible focus rings (v1.2 A11Y-02). Do not remove on the sticky-toast change.
- Error-vs-warn is conveyed by **icon + copy**, not color alone (`err`/`warn` glyphs present) — WCAG 1.4.1.
