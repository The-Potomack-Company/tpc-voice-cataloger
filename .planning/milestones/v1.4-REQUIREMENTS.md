# Milestone v1.4 Requirements — Photo Notes

**Status:** PLANNED 2026-06-10 (intent locked with user 2026-06-10)

v1.4 is a **feature milestone**: cataloging from photographed handwritten notes.
Unlike v1.3 (defect/quality-driven, no REQ set), v1.4 carries a formal requirement
set. IDs use the `PHN-NN` prefix.

## Locked decisions (do not relitigate in phase planning)

| # | Decision |
|---|----------|
| 1 | Vision/OCR via Gemini through the **existing** tpc-ai-proxy Cloud Run service and existing TPC key. No new API enablement, no new billed resources. Needs beyond that = BILLING items for user triage, never provisioned by an agent. |
| 2 | Batch + auto-segment: N photos → one process action → model segments into M items and fills fields per item. |
| 3 | Review gate: OCR output lands as **drafts**; reviewer confirm/edit/discard is the only path to a real session item. |
| 4 | Ships independent of the continuous-mode rework (D-050). Reuse single-shot patterns; do not touch the dormant continuous pipeline. |

## Requirements

### Capture (Phase 46)

- **PHN-01** — From the session detail screen of an active session (house or sale mode), the cataloger can photograph multiple pages of handwritten notes with the rear camera and see them as an ordered page list with thumbnails.
- **PHN-02** — Captured pages persist locally (Dexie) across navigation, tab kill, and offline periods; pages can be reordered, retaken, and deleted before processing. Capture works offline; processing requires connectivity and says so.

### Vision / segmentation (Phase 47)

- **PHN-03** — One process action sends the page batch to Gemini via the existing tpc-ai-proxy (Supabase-JWT auth, `gemini-2.5-flash`, `temperature: 0`, structured `responseSchema`) and produces M draft items with catalog fields filled from the handwriting.
- **PHN-04** — The model returns per-field confidence. Fields below the confidence threshold land **blank and flagged for review** — never a guessed value. The raw OCR text of the source pages is preserved per draft for the reviewer.
- **PHN-09** — Processing failures surface via the shared ErrorToast path and are retryable without creating duplicate drafts (idempotent per process action).
- **PHN-10** — Note pages and drafts are server-persisted (session-scoped RLS) so review can resume cross-device and an audit trail exists. *(Page-blob durability scope is a Phase 47 wave-0 decision — see RISKS R-4.)*

### Review queue (Phase 48)

- **PHN-05** — Drafts live in a review queue on the session detail screen; they are invisible to every existing items consumer (item list/counts, AI queue, export JSON, extension import) until promoted.
- **PHN-06** — Each draft can be edited field-by-field, confirmed (promoted to a real item via the existing creation path), or discarded. No auto-promotion exists.
- **PHN-08** — Sale mode: `receipt_number` always requires explicit reviewer acknowledgment before promotion; duplicate receipts within the session surface the named duplicate-receipt error and block promotion until resolved.

### Contract alignment (Phase 49)

- **PHN-07** — Items promoted from drafts are indistinguishable from manually created items in the export JSON and the extension import path; a regression test pins this. Field-level gaps OCR cannot reliably satisfy are documented and handled by the review gate, not papered over.
