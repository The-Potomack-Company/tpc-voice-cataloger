    - Test: drainQueue processes max 4 items concurrently (concurrency limit)
    - Test: drainQueue retries an item twice on failure, then marks as "failed"
    - Test: drainQueue stops processing if navigator.onLine becomes false mid-drain
    - Test: concurrent drainQueue calls are blocked by mutex (second call returns immediately)

  <action>
Create `src/services/offlineQueue.ts`:

1. Module-level `let draining = false;` mutex flag.

2. `CONCURRENCY = 4` and `MAX_RETRIES = 2` constants.

3. `QueuedItem` interface: `{ id: number; itemType: "house" | "sale"; createdAt: Date }`

4. `getQueuedItems()`: Query `db.houseVisitItems.where("aiStatus").equals("queued").toArray()` and same for `saleItems`. Merge both arrays with itemType tag. Sort by `createdAt` ascending (FIFO). Return `QueuedItem[]`.

5. `findAudioForItem(itemId: number)`: Query `db.audio.where("itemId").equals(itemId).toArray()`. Return the highest `id` audio record (most recent). Return `null` if no audio.

6. `processWithRetry(item: QueuedItem)`:
   - Find audio via `findAudioForItem(item.id)`. If null, mark item as failed and return.
   - Loop up to MAX_RETRIES + 1 attempts:
     - Check `navigator.onLine` before each attempt; if offline, return (leave as queued)
     - Call `processAudioWithAi(audioId, item.id, item.itemType)`
     - On success, return
     - On error: if last attempt, item is already marked "failed" by processAudioWithAi. If not last attempt, reset aiStatus to "queued" before retrying (because processAudioWithAi sets "failed" on any error).

7. `drainQueue()`:
   - If `draining` is true, return immediately
   - Set `draining = true`
   - try: get all queued items, process in batches of CONCURRENCY using `Promise.allSettled`
   - Before each batch, check `navigator.onLine` -- break if offline
   - finally: set `draining = false`

8. Export: `drainQueue`, `getQueuedItems` (for testing and potential UI use).

Add tests to `src/tests/offline-queue.test.ts` (append to existing file from Task 1). Mock `processAudioWithAi` from gemini.ts using vi.mock. Use fake-indexeddb for Dexie. Create test items with aiStatus="queued" in both tables, verify drain behavior.

For the retry test: mock processAudioWithAi to throw twice then succeed -- verify item ends up as "done" (or mock it to throw 3 times and verify it stays "failed").

For concurrency test: track concurrent calls using a counter that increments on entry and decrements on exit of the mocked processAudioWithAi. Assert max concurrent never exceeds 4.

For the mid-drain offline test: after first batch starts processing, set navigator.onLine to false. Verify remaining items are not processed.
