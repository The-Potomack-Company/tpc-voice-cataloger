# v1.2 UI Overhaul — Follow-ups

Items discovered during Phases 23–30 + the Phase 31 mockup-completion sweep
that are out of scope for this milestone but should be picked up later.

## Phase 31 sweep — deferred items

- **Paired Sale-H / House-S two-column tile grid.** The original mockup
  shows sessions inside a date-group as a 2-column grid mixing Sale/House
  tiles side-by-side. The current implementation renders them as a single
  vertical list of tiles inside one `tpc-card` container with dividers (so
  date grouping + accent-wash tone still land). The 2-column layout was
  deferred because the existing session list logic (assignee groups,
  collapsible sections, swipe-to-delete) is denser than the mockup and the
  2-column grid hurts long-name readability on narrow viewports. Worth
  revisiting once the admin grouping is rethought.
- **Recording "back" route from the dedicated mockup screen.** The
  mockup's SCREEN-02 shows a standalone recording surface with a back
  button + dots menu. The current app surfaces recording inline on the
  ItemEntry page; the three-stat strip + waveform + record button land
  there, but the dedicated route isn't planned because the app's flow has
  a per-item record button, not a session-level record screen.
- **Sticky-header sync action on specialist viewport.** Currently the
  Sync button only renders for admins (it triggers an admin-only Export).
  Specialists get the "Submit for Review" CTA below the header. Worth a
  follow-up if specialists ever need a parallel sync action.
- **Walkthrough screen-by-screen Playwright capture.** The sweep added a
  `mockup-fidelity.spec.ts` that captures Login, NewSession, Sessions,
  Settings, AccountManagement. Walkthrough capture is auth-gated +
  state-machine-y; left as a follow-up if a visual-regression baseline is
  needed.

## Component library

- **Storybook / /dev/ui playground.** LIB phase success criterion #4 calls for a visual playground page rendering every primitive variant. The `lib.test.tsx` suite proves every variant renders with the right class names, but a dedicated playground page would help designers + QA review pixel-faithful states (hover, focus, disabled). Suggested path: `src/pages/DevUI.tsx` mounted at `/dev/ui` behind an `import.meta.env.DEV` guard.
- **Inter `<link rel="preload">` for the Inter latin WOFF2.** Deferred from Phase 23 D-08 because emitting a hash-stable href requires extra Vite plugin config (`vite-plugin-static-copy` or a custom resolver). `font-display: swap` already covers FOUT acceptably; preload would just shorten the swap window.

## Screens

- **Walkthrough theme-toggle step.** Phase 25 success criterion #3 (walkthrough step pointing out the new toggle) is partially deferred — the theme picker ships in Settings and the toggle works app-wide, but the walkthrough overlay restyle (Phase 28) preserved the existing step structure. A new "Theme" walkthrough step can be added in a focused follow-up using the existing per-step seen-state model.
- **Live waveform on the recording screen** uses the existing Web Audio analyser if available. If browsers without `AnalyserNode` need a static fallback, this is already covered by the reduced-motion guard, but a more graceful alternative could be a low-effort animated svg.

## Tests / tooling

- **Bundle-size guard for @fontsource imports.** Phase 23 chose not to ship a CI guard. Worth adding once we see real-world drift.
- **Font-family sync test** between `tokens.css` and `@fontsource` registered family names. Cheap; we just didn't need to in v1.2 because the tokens are user-locked.

## A11y

- **Contrast pairs that miss WCAG AA** are documented in `.planning/v1.2-known-issues.md` (Phase 30). Tokens are user-locked so we flag rather than fix; a future phase can revisit if the team agrees to adjust.
